AI Workbench 用户操作指南

从安装到精通，全面掌握你的本地 AI Agent 工作台 —— 不只是聊天，更是自主执行

版本 v0.1.0 14 内置工具 · Agent 模式 · 5 层安全防护 适用：Windows / macOS / Linux
1

什么是 AI Workbench

AI Workbench 是一款本地优先的桌面 AI Agent 工作台——不只是聊天，更能自主浏览项目、修改代码、执行命令。

核心特性

🔒

数据完全在本地

所有聊天记录、配置、密钥都保存在你电脑的 SQLite 数据库中，不经过任何云端服务。

🔌

连接多种大模型

支持 DeepSeek、OpenAI、Claude、通义千问、Kimi、Ollama、LM Studio 等 9+ 服务商，以及自定义端点。

🤖

Agent 自主执行

AI 可以自主浏览项目目录、搜索代码、修改文件、执行命令——多步骤连续工作，无需人工干预每一步。

🧠

工作区记忆

每个项目有独立的持久化记忆，AI 记住你的编码规范、项目架构和长期偏好。

不只是聊天

💡 内置 Skill 和 Tool 系统

AI Workbench 内置 14 个工具和 3 个技能，让 AI 能够：

  • 读取你的本地文件、浏览项目目录、搜索代码
  • 执行授权命令（如 npm run build）
  • 在确认后直接修改代码文件
  • 自动运行测试和构建验证
💡
一句话理解：普通 AI 聊天 = 你问它答；AI Workbench = 你说目标，AI 自己看项目、搜代码、改文件、跑验证、报告结果。

安装与启动

AI Workbench 支持 Windows、macOS 和 Linux 三大平台。

系统要求

操作系统最低版本安装包格式
🪟 WindowsWindows 10 / 11.exe（NSIS 安装程序）
🍎 macOSmacOS 11+.dmg
🐧 Linux主流发行版.AppImage

安装步骤

1

下载安装包

前往 AI Workbench 官方网站，选择与你操作系统对应的安装包进行下载。

2

运行安装程序

  • Windows：双击 .exe 文件，按照安装向导提示完成安装。
  • macOS：打开 .dmg 文件，将应用图标拖入「Applications」文件夹。
  • Linux：赋予 .AppImage 文件执行权限后直接运行。
3

启动应用

安装完成后，从开始菜单（Windows）、Launchpad（macOS）或应用目录（Linux）中启动 AI Workbench。首次打开将显示欢迎界面，并自动初始化 SQLite 数据库和内置 Skill。

✅
首次启动需要初始化数据库、运行迁移、创建内置 Skill，可能需要 2-5 秒。后续启动通常 1-2 秒。

卸载说明

⚠️
Windows 卸载：卸载器会询问是否同步删除本地数据（聊天记录、模型配置、加密密钥、数据库文件）。你可以选择保留，重新安装后数据依然存在。

建议卸载前手动备份 app.db 文件（参见第 15 章数据存储部分），或在设置中点击「清理会话历史」后再卸载。

配置步骤

1

打开模型配置

点击左侧导航栏的「模型配置」或在 Chat 页面顶部点击模型选择器中的「配置模型」。

2

选择 AI 服务商

3

开始聊天

配置好 AI 模型后，你就可以开始和 AI 对话了。

发送消息

1

进入聊天页面

点击左侧导航栏的「Chat」（💬 图标）。

2

新建对话

点击会话列表顶部的「＋」按钮。对话以「新对话」为临时标题，发送第一条消息后标题自动取前 20 个字符。

3

输入并发送

在底部输入框中输入内容，按 Enter 发送（Shift+Enter 换行）。输入框底部有可拖动调整手柄，往上拖可扩大输入区域（80px-260px）。

流式输出

消息发送后，AI 的回复会像打字一样逐字显示（流式输出）。在回复过程中你可能看到：

状态显示内容
等待首个 Token输入区域上方出现加载动画
模型正在思考可折叠的「模型正在思考」面板（部分模型支持 reasoning）
本地工具执行中显示「正在读取文件...」或「正在执行命令...」
正在生成回复文字逐字出现
💡
点击输入框右侧的 停止 按钮（■），正在生成的消息会中止，显示「已取消」状态。

错误处理

如果消息发送失败（API Key 无效、网络不通等）：

  • 页面顶部出现红色错误提示条，显示具体原因
  • 点击 重试 按钮可重新发送上一条用户消息
  • AI 回复失败的消息显示红色边框和错误图标

滚动与「回到底部」

↓ 自动滚动

对话内容较多时，消息列表自动滚动到最新消息。手动往上滚动查看历史后，右下角出现 ↓ 浮动按钮，点击快速回到底部。

工作区（Workspace）

工作区让你能将对话按本地项目文件夹进行分组管理，是 Agent 模式和工具系统的基础。

什么是工作区

📁 工作区 = 一个本地文件夹 + 属于它的对话 + 项目记忆

创建工作区后：

  • 该工作区下新建的对话自动关联到此工作区
  • 工具（如 list_dir、read_file、command_runner）的操作范围限定在工作区根目录内
  • Agent 模式会自动加载工作区的项目上下文（README、package.json、git status 等）
  • 工作区记忆自动注入到每次 Agent 运行中
✅
选择「全局」（不进入任何工作区）时，可创建不和任何文件夹关联的普通对话，所有对话照常保存。

创建与切换工作区

1

创建工作区

在 Chat 页面顶部的工作区选择器中，点击「管理工作区」或「新建工作区」，选择一个本地文件夹作为工作区根目录。工作区名称默认使用文件夹名称。

2

切换工作区

在 Chat 页面顶栏的工作区下拉框中选择一个工作区。会话列表只显示属于该工作区的对话，工具操作默认以工作区根路径为基准。

3

重命名与删除

在工作区管理界面点击铅笔图标重命名（只改数据库记录，不改磁盘文件夹）。点击垃圾桶图标删除工作区——会同时移除此工作区下的所有对话记录和工具日志，但不会删除磁盘上的实际文件夹。

工作区路径失效处理

如果工作区对应的本地文件夹被移动、重命名或删除：

  • Chat 页面顶部显示红色警告提示，包含原工作区名称和磁盘路径
  • 可选择「重新选择工作区」重新绑定到新文件夹
  • 或选择「转为普通对话」将当前对话从工作区解绑
8

消息编辑与重新生成

发错消息或对回复不满意？随时修改和重新生成。

编辑消息（重新提交）

1

悬停在你的消息上

将鼠标悬停在你自己的消息上，点击 编辑 按钮（✏️）。

2

修改并重新提交

修改内容后点击 重新提交 或按 Enter。

⚠️
注意：编辑后，当前消息之后的所有消息（包括 AI 回复）会被删除，AI 基于修改后的内容重新生成回复。这是「从此处重新对话」的行为。

重新生成 AI 回复

将鼠标悬停在最后一条 AI 回复上，点击 重新生成 按钮（🔄）。

💡
只能重新生成最后一条 AI 回复。要修改更早的对话，请用「编辑」功能修改对应的用户消息。

复制 AI 回复

将鼠标悬停在 AI 回复上，点击 复制 按钮（📋），Markdown 格式的完整内容会复制到剪贴板。复制成功后按钮短暂显示 ✔️。

9

导出 Markdown

单条回复或整个对话都可以导出为 Markdown 文件。

导出单条 AI 回复

📥 导出步骤

鼠标悬停在 AI 回复上，点击 导出 Markdown 按钮（📥），在弹出的原生保存对话框中选择路径。默认文件名：AI_Reply_<日期>.md

导出内容包含：标题、生成时间、模型信息、使用的 Skill 名称（如有）、完整回复内容。

导出完整会话

📋 导出整个对话

在 Chat 页面顶部，点击 导出当前会话 按钮，选择保存路径。默认文件名：Conversation_<日期>.md

导出内容包含：

  • 每条用户消息（## User）
  • 每条 AI 回复（## Assistant），含模型信息
  • 工具调用审计摘要表格（## 工具调用记录）：时间、工具名、状态、耗时、摘要
✅
工具输出只导出摘要，不导出完整文件内容。导出通过 Electron 原生对话框完成。

技能（Skills）

Skill 是预设的提示词模板，激活后 AI 会按照特定的风格和格式来处理你的输入。

内置 Skill

应用首次启动时自动创建三个内置 Skill：

Skill ID名称作用触发关键词
text-summary文本总结总结文本内容总结、摘要、提炼、概括
daily-report日报生成将工作内容整理成日报日报、今日工作、工作汇报、工作总结
wechat-rewrite公众号改写改写成公众号文章风格公众号、改写、推文、文章优化

在聊天中使用 Skill

输入框附近有一个 Skill 选择器下拉框，提供三种模式：

模式选择器值行为
自动匹配__auto__（默认）根据消息内容匹配关键词，或根据附件文件类型匹配
不使用 Skill__none__不附加任何 Skill
手动选择具体 Skill 名称强制使用指定 Skill，不论内容是否匹配
💡
一次性上下文：Skill 只在发送当前这条消息时生效。发送后选择器自动切换回「自动匹配」。可以在同一对话中用不同 Skill 处理不同问题。手动选择的优先级最高。

Skill 懒加载机制

⚡ 按需加载，节省上下文

当 Skill 通过自动匹配触发时，AI 先收到 Skill 的概要卡片（名称、描述、触发关键词、工具列表），而非完整 prompt。AI 判断需要使用完整指令时，调用 load_skill 工具加载完整的 system prompt。

✅
手动选择的 Skill 不受此限制——手动选择时直接使用完整 Skill prompt。

创建自定义 Skill

1

进入 Skill 管理页面

点击侧边栏的「Skill」（🔨 图标），切换到「已安装」标签页。

2

创建 Skill

点击「添加 Skill」→「创建 Skill」，在弹出对话框中填写名称、描述和指令（用自然语言描述 AI 应该扮演什么角色）。

技能名称：代码审查员
描述：你是一个严格的代码审查员，检查代码中的潜在问题。
指令：
你是一位资深代码审查员。
请审查用户提供的代码，从以下维度分析：
1. 功能正确性
2. 安全漏洞
3. 性能问题
4. 代码可维护性
对于每个问题，提供具体的代码示例和改进建议。

导入 Skill

支持从外部导入 Skill：

  1. 点击「添加 Skill」→「导入 Skill」
  2. 在文件选择器中选择一个包含 SKILL.md 的文件夹，或一个单独的 .md / .markdown 文件
  3. 导入器解析文件的 YAML 头信息（frontmatter），提取名称、描述、Prompt 等
---
id: my-skill
name: 我的技能
description: 技能描述
trigger:
  keywords:
    - 关键词1
    - 关键词2
  fileTypes:
    - txt
    - md
tools:
  - txt_reader
  - read_file
  - search_text
output:
  format: markdown
---

# System Prompt

这里是系统提示词...

# User Template

这里是用模板，可以用 {{input}} 占位，工具结果用 {{tool_result}} 占位。
💡
如果导入的 Skill ID 已存在，系统自动重命名为 原ID-copy 或 原ID-copy-N。

启用/禁用与删除

  • 开关：在 Skill 卡片上切换启用/禁用。禁用的 Skill 不出现在选择器中，也不会被自动匹配
  • 删除：点击垃圾桶图标 → 确认删除。删除操作不可恢复，但不影响导入源文件夹

工具（Tools）详解

AI Workbench 内置 14 个工具，让 AI 能读取文件、浏览目录、搜索代码、修改文件、执行命令。每个工具都有明确的风险等级。

风险等级体系

等级含义示例
🟢 safe纯只读操作，无副作用读文件、列目录、搜索、查看 git
🟡 medium写工作区内文件，有限副作用创建文件/目录、应用 patch
🔴 high执行命令、写入记忆等需确认的操作执行命令、更新记忆
✅
核心设计原则：文件操作需授权、命令需经确认、按风险分级、结果脱敏、日志不保存完整内容（文件读取只保存元数据和 500 字预览）。

14 个内置工具一览

#工具名分类风险功能

Agent 模式

Agent 模式是本应用的高级工作方式——AI 可以像一位自主的开发者一样多步骤连续工作。

什么是 Agent 模式

🤖 自主多步骤执行

与普通聊天模式（一问一答）不同，Agent 模式下 AI 可以：

  • 自主多步骤执行：连续调用多个工具，每步结果影响下一步决策
  • 主动浏览项目：通过只读工具自己找到需要的信息，不需要你手动选择文件
  • 修改代码：通过 apply_patch 在用户确认后直接修改文件
  • 创建文件和目录：通过 create_file 和 create_directory
  • 自动验证：修改代码后自动运行 npm test / npm build
  • 文件保护：每次写入前自动创建 checkpoint（文件备份）
💡
一句话对比：
• 普通模式：你选文件 → AI 读 → AI 回答（一问一答）
• Agent 模式：AI 看项目结构 → 自己搜索 → 读相关文件 → 生成修改 → 你确认 → 应用修改 → 自动验证 → 报告结果

Agent 运行流程

用户发送消息
↓
请求分类（判断是否为 workspace_task / project_analysis / tool_task）
↓
加载工作区上下文（README、package.json、git status 等）
↓
加载工作区记忆（如有）
↓
注入 Skill（自动匹配时为概要卡片，手动选择时为完整 prompt）
↓
LLM 调用 #1 → AI 思考 → 决定调用工具
↓
执行工具 #1 → 工具结果返回给 AI
↓
LLM 调用 #2 → AI 基于结果继续思考 → 调用更多工具
↓
...（多步循环，直到任务完成或达到安全限制）
↓
apply_patch → 用户确认 → 创建 checkpoint → 写入文件
↓
自动验证：运行 npm test / npm build
↓
LLM 最终调用 → 生成最终回答（附修改摘要、文件列表、验证结果）

安全停止机制

Agent 运行有严格的安全限制，防止无限循环：

限制项默认值说明
墙钟时间10 分钟单次 Agent 运行的总时长上限
总工具调用40 次单次运行的总工具调用次数上限
read_file 预算20 次文件读取工具的单独预算
search 预算20 次搜索工具的单独预算
list_dir 预算20 次目录列出工具的单独预算
重复调用上限3 次同一工具连续调用相同参数的最大次数
无进展轮次3 轮连续无有效进展的最大轮次
总步骤数12 步单次 Agent 运行的最大步骤数

重放防护

高风险工具（command_runner、apply_patch、create_file、create_directory）具有重放防护——如果 AI 尝试用完全相同的输入参数再次调用同一个工具，系统会拒绝执行，防止 AI 陷入无效循环。

自动验证系统

Agent 通过 apply_patch 修改文件后，系统自动运行验证：

  1. 检查工作区是否有 package.json
  2. 如果有 test script → 执行 npm run test（超时 5 分钟）
  3. 如果有 build script → 执行 npm run build（超时 5 分钟）
  4. 验证结果追加到 AI 的最终回复中
本次修改摘要：
- 修复了登录页面的样式问题

文件列表：
- src/pages/Login.tsx
- src/styles/login.css

验证结果：
- npm run build：通过
- npm run test：未通过：1 test failed

未验证原因：
- 测试用例期望值未更新

Agent 检查点

每次 apply_patch 写入文件前，系统自动创建检查点（checkpoint）：

  • 保存内容：文件的完整内容（修改前版本）+ SHA-256 哈希
  • 存储位置：agent_checkpoints 数据库表
  • 用途：如果修改结果不理想，可从检查点恢复

Agent 步骤时间线

Agent 模式下，每条 AI 回复下方显示可展开的「Agent 步骤」面板：

  • 步骤编号：#0, #1, #2...
  • 步骤类型：llm_call（LLM 调用）、tool_call（工具调用）、tool_result（工具结果）
  • 执行状态：running / success / failed / skipped
  • 工具名称和执行耗时（毫秒级）
  • 详细信息：展开后可查看工具调用的参数和返回结果预览

Ask 与 Craft 模式

Agent 模式有两种子模式：

维度Ask（询问）Craft（构建）
行为强制使用 legacy 模式，一问一答当模型支持 tool calls 时使用完整 Agent 模式
适合场景简单问答、代码咨询、理解代码代码修改、项目分析、多步操作
文件访问手动选择单个文件AI 自主浏览工作区文件
工具调用一次性调用多步骤连续调用
代码修改❌ 不支持✅ 支持（需确认）

工作区记忆不加载自动注入

常见问题与故障排除

点击问题展开查看解答。

最常见的几个原因和解决方法：

  • 401 / 403：API Key 无效或过期 → 重新去服务商官网获取新的 Key
  • 404：请求地址填错了 → 检查「请求地址」是否和官方文档一致
  • 超时 / fetch failed：网络不通 → 检查网络；如用本地模型确认服务已启动
  • model not found：模型名称有误 → 检查模型名称大小写和版本号是否准确

检查清单：

  1. 是否设置了环境变量 AI_WORKBENCH_ENABLE_AGENT_MODE=1？（需重启应用）
  2. 当前是否处于 Craft 模式？（Ask 模式下强制 legacy）
  3. 当前模型是否支持 tool calls？（openai-chat-completions 或 anthropic-messages 格式）
  4. 当前对话是否关联了工作区？（部分 Agent 功能需要工作区上下文）

检查清单：

  1. 设置页面中是否已打开 command_runner 开关？
  2. 是否已添加至少一个授权工作目录？
  3. 命令的 cwd 是否在授权目录范围内？
  4. 命令是否触发了危险命令拦截（14 种模式）？
  5. 是否在确认弹窗中点击了「允许」？

可能的原因：

  1. 不在工作区对话中——apply_patch 只能在关联了工作区的对话中使用
  2. 不在 Agent 模式——需要启用 Agent 模式
  3. 目标文件不存在——apply_patch 只能修改已存在的文件
  4. 补丁和目标内容不匹配——文件在 AI 生成补丁后被外部修改过
  5. 目标文件在工作区外部——修改目标必须在工作区目录内

可能的原因：

  1. 不在工作区对话中——update_memory 需要工作区上下文
  2. 内容包含敏感信息——记忆不能包含 API Key、Token、密码等
  3. 内容看起来像聊天记录——系统拒绝保存对话格式的内容
  4. 内容过长——单条记忆不超过 4000 字符

检查清单：

  1. 消息中是否确实包含 Skill 的触发关键词？
  2. 对应的 Skill 是否已启用？
  3. 选择器是否处于「自动匹配」模式（而非「不使用 Skill」）？

如果自动匹配不稳定，可以直接在选择器里手动选技能，优先级最高。

文件读取有几个限制：

  1. 必须通过聊天页面的「选择文件」按钮来选择文件，不能手动输入路径
  2. 只支持 .txt、.md、.markdown 三种格式
  3. 文件大小不能超过 5MB
  4. 文件授权信息在应用关闭后会清除，重新打开应用后需要重新选择文件

默认存储位置：

  • Windows：%APPDATA%/ai-workbench/app.db
  • macOS：~/Library/Application Support/ai-workbench/app.db
  • Linux：~/.config/ai-workbench/app.db

在设置页面可查看和更改存储位置。建议同时备份 app.db、app.db-wal、app.db-shm。

在 Windows 上卸载时，卸载器会询问是否同步删除本地数据（聊天记录、模型配置、加密密钥、数据库文件）。你可以选择保留，重新安装后数据依然存在。

建议卸载前先手动找到 app.db 文件备份，或在设置里点「清理会话历史」再卸载。

# 项目约定

- 使用 TypeScript 严格模式，禁止使用 any
- 所有注释使用中文
- 组件放在 src/components/ 下，页面放在 src/pages/ 下
- 不要修改 src/legacy/ 目录中的代码
- 测试使用 vitest，覆盖率要求 >80%
